# Anthropic Claude / Claude Code の使い方ガイド

Anthropic は Claude を 3 つのプロダクトサーフェスで提供している。下の Claude モデル本体は同じで、変わるのは**周りのワークスペース**だけ。本ノートは、(1) どのサーフェスをいつ使うか、(2) Claude Code から成果物を出すときの形式（HTML/Markdown）の選び方、を 2 本の Anthropic 公式記事から整理したもの。

## 3 つのサーフェスの使い分け

[Founder's Playbook](../sources/anthropic/docs/the-founders-playbook.md) の整理を元にした使い分け表：

| タスクが…… | 使うべきサーフェス | 理由 |
|---|---|---|
| 質問、書き直し、素早いブレスト | **Chat** (claude.ai) | 速い、会話的、セットアップ不要。既に開いているアプリを離れずに使える |
| リサーチ、分析、ファイル・システムから組み上げる完成品 | **Claude Cowork** | フォルダアクセス、コネクタ、Skills、スケジュール実行。ドキュメント・デッキ・スプレッドシート等の生成 |
| ソフトウェアの執筆・テスト・出荷 | **Claude Code** | コードベースアクセス、diff、git 統合、ローカル/IDE/サンドボックス・クラウド環境 |

### 各サーフェスの代表的なユースケース

- **Chat**: 投資家メモから 1 文の要点を引く、取締役会前に主張を健全性チェックする、長い Slack スレッドの要旨を掴む
- **Claude Cowork**: 顧客電話書き起こしフォルダをテーマ別 findings ドキュメントに変える、ベンダーサイトから競合状況を組み上げる、接続ツールから週次 KPI ブリーフを月曜朝に共有フォルダに drop
- **Claude Code**: リーンチームが成長するコードベースで機能を出荷する、MVP 時代のレガシーコード移行、追加採用を待たずにプロトタイプから本番へ運ぶ

「**3 つは内部に同じ Claude を共有する。変わるのはその周りのワークスペースだ。**」（プレイブック原文より）

## Claude Code 出力の選び方：HTML vs Markdown

[Anthropic ブログ「HTML の不合理なまでの有効性」](../sources/anthropic/blog/2026-05-20-unreasonable-effectiveness-of-html.md) より、Claude Code チームは Markdown より HTML を好む傾向に移行している。

### なぜ HTML か

| 観点 | Markdown | HTML |
|---|---|---|
| 情報密度 | テキストとリスト中心 | 表・SVG・CSS・スクリプト・絶対位置・画像など何でも |
| 読みやすい上限 | 約 100 行 | 上限なし（タブ・モバイル対応・リンクで構造化可） |
| 共有のしやすさ | ブラウザでネイティブには描画されない | アップロードしてリンクで共有可 |
| インタラクション | 静的 | スライダー、ノブ、「JSON でコピー」「プロンプトでコピー」等の export |

トークン効率は Markdown が上だが、**Opus 4.7 の 1M 文脈窓では HTML の追加トークンは無視できる**程度、というのが記事の主張。

### HTML を選ぶべきタスクの型

1. **仕様・計画・探索** — 複数案を 1 画面でグリッド比較、データフロー図、モックアップ付きの実装計画
2. **コードレビュー** — diff にインライン注釈、findings を重大度で色分け
3. **デザイン・プロトタイプ** — アニメーション・パラメータをスライダーで調整できる試作
4. **レポート・説明** — ソース横断（Slack/コードベース/git）の要約、SVG 図解、スライドショー
5. **カスタム編集 UI** — ドラッグ可能なカード、フォーム編集、live preview 付きのプロンプト・エディタ（最後に export ボタンを必ず付ける）

### 始め方

「`make an HTML file`」「`make an HTML artifact`」と頼むだけ。主要な要件は**そのアーティファクトの目的と用途を知っている**こと。

## Claude Code を本格的に使うときに揃えるもの

[Founder's Playbook の MVP 章](../sources/anthropic/docs/the-founders-playbook.md) で繰り返し強調されている、Claude Code を「単発生成器」ではなく「永続的協働者」として使うためのお作法：

1. **アーキテクチャ・コンテキスト・ドキュメント** を最初に作る。Claude Code が本番コードを 1 行書く前に、Claude（Chat）と一緒に、原則・避けるべき依存・許容したトレードオフを定義する。
2. **`CLAUDE.md` に保存する** — Claude Code がディレクトリ内で起動すると自動で読まれるプロジェクト固有の指示・コンテキスト（事実上の「永続メモリ」）。
3. **スコープ・ドキュメント** を別途用意 — 製品が何をするか、**意図的にしないこと**、新機能を追加するに値する実ユーザーからのエビデンス（機能追加基準）を明文化。スコープクリープの強制機能になる。
4. **各セッションを揃える** — セッション開始時にスコープ＋`CLAUDE.md` を読ませ、終了時に決定事項・導入した仮定を `CLAUDE.md` に追記する。「5 分のドキュメント化はアーキテクチャ・ドリフトに対する安価な保険」。

## やってはいけないこと

プレイブックが警告する典型的な失敗モード：

- **検証を「作ること」で代替する**：Claude Code でプロトタイプが楽に出るので、ユーザー検証をスキップして「作ったから正しいはず」と思い込む。プロトタイプは検証の道具であって、検証の証拠ではない。
- **早期スケーリング**：構築摩擦がほぼゼロのため、ビジネス側の検証より先に実行を伸ばしてしまう。「システム内の知性はあなたのもの」——感覚形成 (sense-making) を構築より先行させ続ける。
- **確証バイアスの増幅**：「自分の仮説を支持して」と AI に頼むと、説得力ある支持資料が出てくる。同じツールを**反対方向**（圧力テスト・悪魔の代弁者）にも使うこと。
- **セキュリティの後回し**：エージェント型コーディングは動くコードを生むが本来的に安全なコードは生まない。ユーザーが触れる前のセキュリティレビューが「最小の責任ある閾値」。
- **CLAUDE.md なしの長期運用**：セッションごとに基礎決定がドリフトし、後から壊れる。
