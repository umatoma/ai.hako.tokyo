# 効く CLAUDE.md の書き方と育て方

`CLAUDE.md` は Claude Code がディレクトリ内で起動すると**自動で読み込む**ファイルで、事実上「プロジェクトの永続メモリ」として機能する。書き方次第でセッション間の一貫性とアーキテクチャ・ドリフト耐性が劇的に変わる。

> 出典: Founder's Playbook MVP 章の「アーキテクチャ・コンテキスト・ドキュメントを最初に作り `CLAUDE.md` に保存する」「各セッションを揃える（開始時に読ませ、終了時に追記）」「5 分のドキュメント化はアーキテクチャ・ドリフトに対する安価な保険」。

## どんな情報を入れるか・入れないか

| 入れる | 入れない |
|---|---|
| 主要コマンド（build / test / lint / dev サーバ起動・単体テスト実行方法） | 自明な開発作法（「テストを書け」「エラーハンドリングしろ」） |
| big picture アーキテクチャ（**複数ファイル読まないと分からない**部分） | ファイル構造の機械的な列挙（`ls` で分かる） |
| 意図的に避ける依存・許容したトレードオフ | 一般論的なベストプラクティス |
| スコープ・機能追加基準（または `SCOPE.md` への参照） | 揮発性 TODO（コード内 TODO / Issue へ） |
| やってはいけないこと（セキュリティ・運用ルール） | 各セッションの一時メモ |
| セッション開始・終了ルーチン | 解決済みのバグ修正履歴（git log で分かる） |

**判定の鍵**: 「これが無いと次のセッションで Claude が同じ間違いを繰り返すか？」が Yes なら入れる。No なら入れない。

## 推奨セクション構成

新規 `CLAUDE.md` の骨格例:

```markdown
# CLAUDE.md

このファイルは Claude Code (claude.ai/code) がこのリポジトリで作業するときの指針。

## このリポジトリの目的

（1〜3 行）

## 主要コマンド

- ビルド: `npm run build`
- テスト: `npm test`
- 単体テスト: `npm test -- <file>`
- dev サーバ: `npm run dev`

## アーキテクチャ (big picture)

（複数ファイル横断でしか分からない構造だけ。ファイル一覧は書かない）

## 意図的に避ける依存・許容したトレードオフ

- ...

## スコープ・機能追加基準

詳細は [`SCOPE.md`](./SCOPE.md)。要旨: ...

## やってはいけないこと

- ...

## セッション運用

- セッション開始: `SCOPE.md` も読むこと
- セッション終了: このセッションで決定したこと・導入した仮定を本ファイルの該当節に追記
```

## 育て方（セッションごとの差分追記）

`CLAUDE.md` は一度書いて終わりではなく、**セッション終了時に差分を追記する**ことで価値が出る。Founder's Playbook の言う「5 分のドキュメント化」は具体的にはこれ:

- そのセッションで決めた**設計判断**（例:「ユーザーセッションは Redis ではなく DB に持つ。理由: 運用要員 1 名のためインフラを増やさない」）
- 採用した**規約**（例:「日付は ISO8601 文字列で受け渡す。Date オブジェクトは境界でしか使わない」）
- 確定した**避ける依存**（例:「ORM は使わない、SQL を直接書く」）

避けるべきは:

- そのセッション固有の文脈（「今日のレビューで X さんが指摘した」）
- 後で気が変わる可能性のある TODO
- なぜ駄目だったか説明できない感想的な禁則

## アンチパターン

- **長すぎる `CLAUDE.md`**: 数百行になったら、トピック別に分割し（`docs/architecture.md` 等）、`CLAUDE.md` からはリンクだけにする。Claude Code が毎セッション全部読むコスト分の価値があるか考える。
- **コードに書ける情報を `CLAUDE.md` に書く**: 関数のシグネチャ説明・型定義の言い換えなど。コードと docstring に書く。
- **「やれ」だけで「なぜ」が無い禁則**: 「`any` を使うな」だけ書いても破る理由を考えない。Founder's Playbook の発想だと、なぜ＝過去の事故 / 制約 / 強い好み を 1 行添える。
- **規約と現状の混在**: ルール（規約）と「現在のリポにあるもの」（現状）は別ファイルにする。このリポでは [`INDEX.md`](../INDEX.md) が現状を担当している。
- **PR ごとの大改造**: `CLAUDE.md` を頻繁に書き換えると、過去セッションが守っていた前提が崩れる。追記主体・削除は慎重に。

## チェックリスト（セルフレビュー）

書き終えたら次を確認:

- [ ] 「これが無いと次のセッションで同じ間違いを繰り返す」項目だけになっているか
- [ ] 主要コマンドは実際に動くか（コピペして検証）
- [ ] 「避ける依存」「許容したトレードオフ」の**理由**が 1 行ずつ添えられているか
- [ ] スコープと機能追加基準への参照があるか
- [ ] セッション開始・終了のルーチンが書かれているか
- [ ] ファイル構造の機械的な列挙が混ざっていないか（`ls` で分かることは書かない）
- [ ] 200 行を超えていないか（超えたら分割を検討）
