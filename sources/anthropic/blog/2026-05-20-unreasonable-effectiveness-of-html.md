# Claude Code を使う: HTML の不合理なまでの有効性

**著者:** Thariq Shihipar, Member of Technical Staff
**日付:** 2026 年 5 月 20 日
**読了時間:** 約 5 分

---

## はじめに

Markdown は、シンプルさ・可搬性・リッチテキスト能力を理由に、エージェントが人間とやりとりするための支配的なファイル形式になった。しかしエージェントが強力になるにつれて、Markdown は複雑な作業の出力に対してますます制約の大きな形式になりつつある。

Claude Code チームは、出力形式として Markdown ではなく HTML を好み始めている。HTML のほうがリッチな可視化・色彩・図表の生成や、出力の共有のしやすさに適していることが分かったからだ。

---

## なぜ HTML を使うのか？

### 情報密度

HTML は Markdown と比較して、はるかに多くの情報を表現できる。たとえば：

- 表によるタブラー・データ
- CSS によるデザイン・データ
- SVG によるイラスト
- script タグによるコード断片
- JavaScript と CSS を備えた HTML 要素によるインタラクション
- SVG と HTML によるワークフロー
- 絶対位置や canvas による空間データ
- img タグによる画像

> "There is almost no set of information that Claude can read that you cannot efficiently represent with HTML."

「Claude が読めて HTML で効率的に表現できない情報の集合は、ほとんど存在しない。」

### 視覚的明瞭さと読みやすさ

Claude がより複雑な仕事に取り組むにつれて、より大きな仕様書や計画書を生み出すようになる。HTML ドキュメントは、Claude がタブ・イラスト・リンクで視覚的構造を整理できるため、はるかに読みやすい。ドキュメントは異なる閲覧文脈に対応してモバイル・レスポンシブにもできる。

Markdown が読みやすい上限はおおよそ 100 行程度だが、HTML ドキュメントはこの制限を大きく超えて使える。

### 共有のしやすさ

Markdown ファイルは共有しづらい。ブラウザがネイティブではうまく描画しないからだ。HTML ファイルはアップロードしてリンクで共有でき、どこにいる同僚にもアクセス可能になる。これは、仕様書・レポート・PR 説明文をステークホルダーが実際に読む可能性を劇的に高める。

### 双方向インタラクション

HTML はドキュメントとのインタラクティブな関わりを可能にする。デザインやアルゴリズム・パラメータを調整するスライダーやノブを Claude に追加するよう頼める。「JSON としてコピー」「プロンプトとしてコピー」のようなエクスポート機能は、ユーザーをループの中に保ち、フィードバック・サイクルを引き締める。

### データ取り込み

Claude Code は、ファイルシステム、MCP（Slack、Linear など）、Web ブラウザ統合、git 履歴を通じて広範なコンテキストを取り込める——これらは Claude.ai や Claude Design では利用できない能力だ。この広いコンテキストにより、より洗練された HTML アーティファクトの生成が可能になる。

---

## 始め方

Claude に HTML を生成させるためにすべきことは多くない。単に「_HTML ファイルを作って (make an HTML file)_」あるいは「_HTML アーティファクトを作って (make an HTML artifact)_」と頼めばよい。主要な要件は、そのアーティファクトの意図された目的と用途を知っていることだ。

---

## ユースケース

### 仕様、計画、探索

HTML は問題解決のためのリッチなキャンバスとして機能する。Claude Code に、複数の異なるアプローチをブレインストームさせ、異なるオプションの探索を作らせ、モックアップを開発させ、実装計画を生成させる——すべて相互にリンクされた HTML ファイルとして——よう頼める。

**プロンプト例:**

- 「_オンボーディング画面でどの方向に進むべきか決めかねている。レイアウト・トーン・密度を変えて、明らかに異なる 6 つのアプローチを生成し、単一の HTML ファイル内にグリッドで並べて、横並びで比較できるようにしてくれ。_」
- 「_HTML ファイルで詳細な実装計画を作ってくれ。モックアップを必ず含め、データフローを示し、重要なコード断片を加えてくれ。_」

**こんなときに使う:**
- 実装アプローチの探索
- 複数のビジュアル・デザインの同時実験

### コードレビューと理解

HTML を使うと、diff・アノテーション・フローチャート・モジュールをレンダリングできる。HTML を使って、エージェントが書いたコードを理解したり、プルリクエストをレビューしたり、レビュアーに変更を説明したりできる。

**プロンプト例:**

「_この PR のレビューを助けてくれ。それを記述する HTML アーティファクトを作ること。ストリーミング/バックプレッシャー・ロジックに焦点を当てる。実際の diff をインライン・マージン・アノテーション付きでレンダリングし、findings を重大度で色分けしてくれ。_」

**こんなときに使う:**
- プルリクエストのドキュメンテーション作成
- プルリクエストのレビュー
- コード・トピックの理解

### デザインとプロトタイプ

HTML の表現力は、最終実装が他の言語（React、Swift など）をターゲットにする場合でも、デザイン・スケッチに適している。アニメーションやインタラクティブ・コントロールでのアクションを含む、インタラクションのプロトタイピングが可能だ。

**プロンプト例:**

「_チェックアウトボタンのアニメーション・パラメータをいろいろ試せるよう、複数のスライダーとオプションを備えた HTML ファイルを作ってくれ。うまくいったパラメータをエクスポートするコピーボタンも付けてくれ。_」

**こんなときに使う:**
- デザイン・システム・アーティファクトの作成
- コンポーネントの調整
- コンポーネント・ライブラリの可視化
- アニメーションのプロトタイピング

### レポート、リサーチ、学習

Claude Code は、複数のデータソース（Slack、コードベース、git 履歴、インターネット）にわたる情報を統合して、読みやすい HTML レポート、インタラクティブな説明、図に SVG を使ったスライドショー・デッキを作るのに有効だ。

**プロンプト例:**

「_我々のレート・リミッターがどう動いているのか分からない。関係するコードを読んで、単一の HTML 説明ページを作ってくれ：トークンバケットのフロー図、3〜4 個の重要なアノテーション付きコード断片、そして「落とし穴」セクションを含めて。_」

**こんなときに使う:**
- 機能の要約執筆
- 説明資料の生成
- 週次ステータス・レポートの起草
- インシデント・レポートの作成
- SVG イラストや技術図の作成

### カスタム編集インターフェース

要件を記述しにくいときは、Claude に目的特化型の HTML エディタ・インターフェースを作るよう頼む。鍵は、UI 上の修正を共有可能な形式に戻すエクスポート機能（「JSON としてコピー」「プロンプトとしてコピー」）で終わらせることだ。

**プロンプト例:**

- 「_各チケットをドラッグ可能なカードとして、Now/Next/Later/Cut の各カラム間を移動できる HTML ファイルを作ってくれ。最終順序をエクスポートする「Markdown としてコピー」ボタンを追加。_」
- 「_我々のフィーチャー・フラグ設定用の、フォーム・ベースのエディタを作ってくれ。領域でグループ化し、依存関係を表示し、無効化された前提条件について警告すること。_」
- 「_並列エディタを作ってくれ：左に編集可能なプロンプト（変数スロットをハイライト）、右に 3 つのサンプル入力を置き、埋め込まれたテンプレートをライブで再描画する。_」

**こんなときに使う:**
- 項目の並べ替え・トリアージ・バケット分け
- 制約付きの構造化設定の編集
- ライブ・プレビュー付きのプロンプト・テンプレート調整
- データセットのキュレーション
- ドキュメントへの注釈付けとエクスポート
- テキストでは表現しにくい値の選択（色、イージング・カーブ、クロップ領域、cron スケジュール、正規表現など）

## よくある質問

### 効率が悪くないか？

Markdown のほうがトークン数は少ないが、HTML の表現力の高さと読まれる可能性の高さを足し合わせると、全体としては良い出力になる。Opus 4.7 の 1M 文脈窓 (context window) では、トークン使用量の増加はごくわずかだ。

### Markdown はいつ使うのか？

著者はほぼすべてに対して Markdown を使うのをやめており、「自分はおそらく HTML マキシマリスト側に大きく寄っている」と認めている。

### これはあなたが計画を置き換えた方法か？

複数の HTML ファイルが異なる計画段階に役立つ。実装計画・UI 探索ファイル・コンポーネント・デザイン・リストが、単一の巨大計画を置き換える。これらのファイルは継続的な参照および検証コンテキストとして機能する。

---

## Claude のループにとどまる

Markdown よりも HTML を好む根本的な理由は、Claude の仕事と関与を維持することだ。Claude が大きな責任を担うほど、計画を徹底的に読まなくなる。HTML は、受動的な引き渡しを受け入れる代わりに、Claude の選択へのより密接な関与を促進する。

> "I feel more in the loop now than I ever did before."

「これまでよりずっとループの中にいる感覚がある。」
