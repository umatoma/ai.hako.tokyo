# AI による仕様駆動開発: 新しいオープンソースツールキットで始める

**著者:** Den Delimarsky ([@localden](https://github.com/localden))
**公開日:** 2025 年 9 月 2 日
**カテゴリ:** [AI & ML](https://github.blog/ai-and-ml/) / [Generative AI](https://github.blog/ai-and-ml/generative-ai/)

## 概要

コーディングエージェントが強力になるにつれて、あるパターンが浮かび上がってきた: ゴールを記述するとコードのブロックが返ってきて、たいてい… 見た目は正しいけれど、ちゃんとは動かない。この「vibe-coding（雰囲気コーディング）」アプローチは素早いプロトタイプには素晴らしいが、本格的でミッションクリティカルなアプリや既存コードベースに取り組むときには信頼性が低い。

ときにコードがコンパイルされない。ときに問題の一部だけを解決し、本当の意図は外している。スタックやアーキテクチャがこちらの選びたいものと違うこともある。

問題はコーディングエージェントのコード力ではなく、我々のアプローチだ。コーディングエージェントを検索エンジンのように扱っているが、本来は **「文字通り受け取るペアプログラマー」**として扱うべきだ。パターン認識には長けているが、それでも曖昧でない指示が必要。

だから我々は仕様 (specifications) を再考している ― 静的なドキュメントとしてではなく、プロジェクトと共に進化する**生きた実行可能な成果物 (executable artifacts)** として。仕様が共有された真実の源泉 (source of truth) になる。意味が通らなくなったら仕様に戻る。プロジェクトが複雑になったら仕様を磨く。タスクが大きすぎたら分解する。

**Spec Kit** ― 我々の新しい仕様駆動開発向けオープンソースツールキット ― は、GitHub Copilot、Claude Code、Gemini CLI といったツールを使ったコーディングエージェントワークフローに、構造化された仕様駆動開発プロセスを持ち込む。

---

## 💡 仕様駆動開発 (spec-driven development) とは？

コードを書いてからドキュメントを書く代わりに、仕様駆動開発ではまず仕様 (spec) から始める。仕様はコードがどう振る舞うべきかの「契約 (contract)」であり、ツールや AI エージェントがコードを生成・テスト・検証する真実の源泉になる。結果として、推測が減り、驚きが減り、コードの品質が上がる。

---

## Spec Kit による仕様駆動プロセスとは？

[Spec Kit](https://github.com/github/spec-kit) は、仕様をエンジニアリングプロセスの中心に据える。仕様を書いて棚にしまうのではなく、仕様が実装・チェックリスト・タスク分解を駆動する。あなたの主な役割は「舵を取ること」であり、コーディングエージェントが書く作業の大部分を担う。

プロセスは 4 つのフェーズと明確なチェックポイントで動く。だが鍵となる気づきはこうだ: 各フェーズには特定の仕事があり、そのタスクが完全に検証されるまで次へ進まない。

### 1. **Specify（仕様化）**

何を作るか、なぜ作るかを高レベルで記述すると、コーディングエージェントが詳細な仕様を生成する。これは技術スタックやアプリ設計の話ではない。ユーザージャーニー、体験、成功の姿の話だ。誰が使うのか？ どんな問題を解決するのか？ どう操作するのか？ どんな成果が重要なのか？ 創りたい UX をマッピングし、詳細はコーディングエージェントに肉付けさせる。重要なのは、これがユーザーとそのニーズについて学ぶにつれて進化する**生きた成果物 (living artifact)** になること。

### 2. **Plan（計画）**

ここで技術的になる。希望する技術スタック・アーキテクチャ・制約をコーディングエージェントに与えると、包括的な技術計画が生成される。会社が特定技術に標準化していればここで言う。レガシー連携、コンプライアンス要件、性能目標があればすべてここに入れる。複数の計画バリエーションを比較対照することもできる。社内ドキュメントをコーディングエージェントに見せれば、アーキテクチャパターンと標準を計画に直接統合してくれる。結局、コーディングエージェントはゲームを始める前にルールを理解する必要がある。

### 3. **Tasks（タスク）**

コーディングエージェントが仕様と計画を取り、実際の作業に分解する。それぞれが特定のピースを解決する、小さくレビューしやすいチャンクを生成する。各タスクは隔離して実装・テストできる必要がある ― これが重要なのは、コーディングエージェントが自分の作業を検証して軌道を保つ手段になるから ― ほぼ AI エージェントのテスト駆動開発プロセスのように。「認証を作る」ではなく、「メール形式を検証するユーザー登録エンドポイントを作る」のような具体的なタスクを得る。

### 4. **Implement（実装）**

コーディングエージェントがタスクを 1 つずつ（あるいは可能なら並行で）片付けていく。だがここが違う: 千行のコードダンプをレビューする代わりに、開発者は特定の問題を解決する焦点の合った変更をレビューする。コーディングエージェントは「何を作るべきか」を知っている ― 仕様が教えたから。「どう作るか」を知っている ― 計画が教えたから。「何に取り組むか」を正確に知っている ― タスクが教えたから。

重要なのは、あなたの役割は舵取りだけではなく、**検証**でもあること。各フェーズで反省し洗練する。仕様は本当に作りたいものを捕まえているか？ 計画は現実の制約を考慮しているか？ AI が見落とした抜けやエッジケースはないか？ プロセスには明示的なチェックポイントが組み込まれており、次へ進む前に生成物を批評し、ギャップを見つけ、軌道修正できる。AI が成果物を生成し、あなたがそれが正しいことを保証する。

## エージェント型ワークフローでの Spec Kit の使い方

Spec Kit は GitHub Copilot、Claude Code、Gemini CLI などのコーディングエージェントと動く。鍵は、シンプルな一連のコマンドでコーディングエージェントを舵取りし、エージェントが成果物を生成する重作業を担うこと。

セットアップは簡単。まず `specify` コマンドラインツールをインストールする。これがプロジェクトを初期化し、必要な構造を準備する。

```
uvx --from git+https://github.com/github/spec-kit.git specify init <PROJECT_NAME>
```

プロジェクトが初期化されたら、`/specify` コマンドで高レベルプロンプトを与え、コーディングエージェントに完全な仕様を生成させる。プロジェクトの「何 (what)」と「なぜ (why)」に集中し、技術詳細には触れない。

次に、`/plan` コマンドで技術実装計画を作らせる。高レベルの技術方針を与え、アーキテクチャと制約を尊重した詳細な計画をエージェントに生成させる。

最後に、`/tasks` コマンドで仕様と計画を実行可能なタスクのリストに分解させる。コーディングエージェントはこのリストを使ってプロジェクト要件を実装する。

この構造化されたワークフローが、曖昧なプロンプトを「コーディングエージェントが信頼性高く実行できる明確な意図」に変える。

だが、なぜこのアプローチが曖昧なプロンプティングが失敗する場面で成功するのか？

## なぜこれが効くのか

このアプローチが「AI にプロンプトを投げるだけ」より成功するのは、言語モデルの動作についての基本的真実による: 彼らはパターン補完には長けているが、心を読むのは苦手だ。「アプリに写真共有を追加して」のような曖昧なプロンプトは、モデルに何千もの言明されない要件を推測させてしまう。AI は合理的な仮定をするが、その一部は間違っており（しかも、実装の深い段階に入るまで間違いに気づかないことが多い）。

対照的に、明確な仕様、技術計画、焦点を絞ったタスクを先に与えると、コーディングエージェントは明確さを得て全体の有効性が上がる。あなたのニーズを推測する代わりに、何を作り、どう作り、どの順序で作るかを知る。

これが異なる技術スタックを跨いで効く理由でもある。Python、JavaScript、Go で作る場合でも、根本の課題は同じ ― 意図を動くコードに翻訳する。仕様が意図を明確に捉え、計画がそれを技術判断に翻訳し、タスクが実装可能なピースに分解し、AI コーディングエージェントが実際のコーディングを担う。

大きな組織では、これがもう一つ重要な問題を解く: セキュリティポリシー、コンプライアンスルール、デザインシステム制約、連携ニーズなどの要件をどこに置くか。多くの場合これらは誰かの頭の中、誰も読まない Wiki、後で見つけられない Slack 会話に散らばっている。

Spec Kit を使えば、すべてのそれは仕様と計画の中に入り、AI が実際に使える。セキュリティ要件は後付けではなく、初日から仕様に焼き込まれる。デザインシステムは後で接合するものではなく、実装を導く技術計画の一部だ。

このアプローチの反復性こそが力を与えている。伝統開発が早期判断にロックする一方、仕様駆動は方向転換を簡単にする: 仕様を更新し、計画を再生成し、コーディングエージェントに残りを任せる。

## このアプローチが特に効く 3 つの場面

仕様駆動開発は次の 3 つのシナリオで特に有用:

### 1. **グリーンフィールド (zero-to-one)**

新プロジェクトを始めるとき、つい即コーディングしたくなる。だが少しの先行作業で仕様と計画を作っておけば、AI が「よくあるパターンの汎用解」ではなく、本当に意図したものを構築してくれる。

### 2. **既存システムへの機能追加 (N-to-N+1)**

仕様駆動開発が最も強力な場面。複雑な既存コードベースに機能を追加するのは難しい。新機能の仕様を作ることで、既存システムとどう相互作用すべきかを明確にせざるを得なくなる。計画がアーキテクチャ制約を符号化し、新コードが既存プロジェクトに「ネイティブ」に感じられるようにする ― 後付け感がない。これにより継続的な開発が速く安全になる。これを機能させるには高度な「コンテキストエンジニアリング」のプラクティスが必要かもしれない ― これは別途取り上げる。

### 3. **レガシー近代化**

レガシーシステムを再構築する必要があるとき、当初の意図は時間の中で失われていることが多い。Spec Kit の仕様駆動開発プロセスを使えば、本質的なビジネスロジックをモダンな仕様に捉え、計画で新鮮なアーキテクチャを設計し、AI に既存技術負債を引き継がせずゼロからシステムを再構築させられる。

コアの利点は、安定した「何 (what)」と柔軟な「どう (how)」を分離することで、高価な書き直しなしの反復開発を可能にすること。これにより複数バージョンを構築し、素早く実験できる。

## 我々が向かう先

我々は「コードが真実の源泉」から「**意図 (intent) が真実の源泉**」へ移行している。AI とともに、仕様が真実の源泉になり、何が作られるかを決定する。

これはドキュメントが重要になったからではない。AI が仕様を実行可能にするからだ。仕様が自動で動くコードになるなら、仕様こそが「何が作られるか」を決定する。

Spec Kit はその移行を現実のものにする我々の実験だ。このアプローチが単一のツールや企業より大きいから、オープンソース化した。本当のイノベーションはプロセスだ。次にカバーする ― 特に、仕様駆動開発プラクティスをコンテキストエンジニアリングと組み合わせて、AI ツールキットでより高度な能力を作る方法。

そしてあなたにとってどう動き、何を改善できるかをぜひ聞きたい！ 仕様駆動パターンで作っているなら、[体験を共有してほしい](https://github.com/github/spec-kit/issues)。特に興味があるのは:

- **ワークフローをより魅力的で使いやすくする**: 大量のテキストを読むのは退屈になりがち。このプロセスを本当に楽しいものにするには？
- **VS Code 連携の可能性**: このワークフローを VS Code に直接持ち込む方法を探っている。何が最も自然に感じる？
- **複数実装の比較・差分**: 反復と実装間の差分は創造的な可能性を開く。ここで最も価値あるものは？
- **組織での大規模な仕様・タスク管理**: 多くの Markdown ファイル管理は圧倒される。整理と集中を保つには何が役立つ？

我々は、人間の創造性を動くソフトウェアに翻訳するより良い方法を AI を使って探求するあなたの姿を見るのが楽しみだ。
