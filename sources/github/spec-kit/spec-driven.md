# Specification-Driven Development (SDD)

## 力関係の逆転 (The Power Inversion)

何十年もの間、コードこそが王だった。仕様 (specifications) はコードに仕える存在 ― 構築の足場として作られ、「本物の仕事」であるコーディングが始まるとともに捨てられるものだった。我々は開発を導くために PRD (Product Requirements Document) を書き、実装に情報を与える設計書を作り、アーキテクチャを可視化するために図を描いた。しかしそれらは常にコードに従属していた。コードが真実だった。それ以外はせいぜい「善意」だった。コードが信頼できる情報源 (source of truth) であり、コードが前進するにつれて仕様は追いつけなくなる。資産（コード）と実装は同一なので、コードを離れて並行実装を立てるのは容易ではない。

Spec-Driven Development (SDD) は、この力関係を逆転させる。仕様がコードに仕えるのではなく ― **コードが仕様に仕える**。PRD は実装の手引きではなく、実装を生成する源 (source) である。技術計画はコーディングに情報を与える文書ではなく、コードを生み出す精密な定義である。これは漸進的改善ではない。「何がソフトウェア構築を駆動するか」を根本から問い直すものだ。

仕様と実装の間のギャップは、ソフトウェア開発の創世期から続く宿痾だった。よりよいドキュメント、より詳細な要件、より厳格なプロセス ― これらの試みは失敗してきた。なぜならギャップを必然と受け入れているからだ。狭めようとはしても、消すことはなかった。SDD は、仕様と仕様から生まれる具体的な実装計画とを「実行可能 (executable)」にすることで、ギャップを消し去る。仕様と実装計画がコードを生成するなら、そこにギャップはなく、あるのは「変換」だけだ。

これが今、可能になったのは、AI が複雑な仕様を理解・実装でき、詳細な実装計画も作れるようになったからだ。だが構造のない生成は混沌を生む。SDD は、動くシステムを生成できるほど精密で完全かつ曖昧でない仕様と実装計画によって、その構造を提供する。**仕様が主たる成果物 (primary artifact)** となり、コードはその「表現 (expression)」 ― 特定の言語とフレームワークでの ― になる。

この新しい世界では、ソフトウェアを保守することは「仕様を進化させること」を意味する。開発チームの意図は自然言語（「**意図駆動開発 (intent-driven development)**」）、デザイン資産、コア原則、その他のガイドラインで表現される。開発の**共通言語 (lingua franca)** はより高い抽象度へ移り、コードは「ラストマイル」になる。

デバッグは、不正なコードを生む仕様と実装計画を修正すること。リファクタリングは、明瞭さのための再構成。開発ワークフロー全体が、仕様を中心的な情報源として再構成され、実装計画とコードは継続的に再生成される出力となる。アプリへの新機能追加や並行実装の創出 ― 我々が創造的な存在である以上 ― は、仕様を見直し、新しい実装計画を作ることを意味する。このプロセスは 0 → 1, (1', …), 2, 3, N となる。

開発チームは創造性、実験、批判的思考に集中する。

## 実践における SDD ワークフロー

ワークフローはアイデアから始まる ― 多くの場合、曖昧で不完全なアイデアから。AI との反復対話を通じて、このアイデアは包括的な PRD になる。AI は明確化の質問を投げ、エッジケースを特定し、精密な受け入れ基準を定める手伝いをする。従来開発で数日かかった会議とドキュメント作りが、数時間の集中した仕様作業で済む。これは伝統的な SDLC を変容させる ― 要件と設計は離散的なフェーズではなく、継続的な活動になる。これは**チームプロセス**を支援するものだ。チームでレビューされる仕様が表現・バージョン管理され、ブランチで作られマージされる。

プロダクトマネージャが受け入れ基準を更新すると、影響を受ける技術判断が実装計画に自動でフラグされる。アーキテクトがより良いパターンを見つけると、PRD が新しい可能性を反映するように更新される。

仕様プロセス全体を通じて、リサーチエージェントが重要な文脈を集める。ライブラリの互換性、性能ベンチマーク、セキュリティ上の含意を調査する。組織的制約 ― 自社のデータベース標準、認証要件、デプロイポリシー ― が自動的に発見され、あらゆる仕様にシームレスに統合される。

PRD から、AI は要件を技術判断にマップする実装計画を生成する。すべての技術選択に文書化された理由がある。すべてのアーキテクチャ判断は特定の要件にトレースできる。このプロセスを通じて、整合性検証が継続的に品質を高める。AI は仕様の曖昧性・矛盾・欠落を分析する ― ワンタイムのゲートではなく、継続的な洗練として。

仕様と実装計画が十分に安定すれば（「完璧」である必要はない）、コード生成が始まる。初期の生成は探索的かもしれない ― 仕様が実践で意味を成すかを試す。ドメイン概念はデータモデルに、ユーザーストーリーは API エンドポイントに、受け入れシナリオはテストになる。テストシナリオはコードの後ではなく、実装とテストの両方を生む仕様の一部として書かれる ― これにより開発とテストが仕様を通じて統合される。

フィードバックループは初期開発で終わらない。本番環境のメトリクスとインシデントはホットフィックスを誘発するだけでなく、次の再生成に向けて仕様を更新する。性能ボトルネックは新しい非機能要件になり、セキュリティ脆弱性は今後の全生成に影響する制約になる。仕様と実装と運用現実の間の反復的な舞いの中で、真の理解が現れ、伝統的 SDLC が継続的な進化へと変容する。

## なぜ SDD が今、重要なのか

3 つのトレンドが SDD を可能にし、必要にする:

第一に、AI 能力が「自然言語仕様から信頼できる動作コードを生成できる」閾値に達した。これは開発者の置き換えではなく、機械的な「仕様 → 実装」変換を自動化することで開発者の効果を増幅することだ。探索・創造性を増幅し、「やり直し」を容易にし、加算・減算・批判的思考を支援する。

第二に、ソフトウェアの複雑さは指数関数的に増え続けている。現代のシステムは数十のサービス・フレームワーク・依存ライブラリを統合する。これらすべてを当初の意図と整合させ続けるのは、手動プロセスでは難しくなる一方だ。SDD は仕様駆動生成によって体系的な整合を提供する。フレームワークは「人間ファースト」ではなく「AI ファースト」のサポートへ進化するかもしれないし、再利用可能なコンポーネントを中心に再設計されるかもしれない。

第三に、変化のペースが加速している。要件の変化は史上最速。ピボットは例外ではなく当たり前。現代のプロダクト開発は、ユーザーフィードバック・市場・競合圧力に対する素早い反復を求める。伝統開発はこれらを破壊として扱う。各ピボットがドキュメント・設計・コードへの手作業の伝播を要求する。結果として、速度を犠牲にした慎重な更新か、技術的負債を積み上げる迅速で粗い変更かのどちらかになる。

SDD は「もし～なら (what-if/simulation)」の実験をサポートできる: 「もし T シャツをもっと売りたいというビジネスニーズに応えるためアプリを再実装・変更したいなら、どう実装し実験するか？」

SDD は要件変更を「障害」から「通常のワークフロー」に変える。仕様が実装を駆動するなら、ピボットは手動書き直しではなく体系的な再生成になる。PRD のコア要件を変えれば、影響を受ける実装計画は自動更新される。ユーザーストーリーを修正すれば、対応する API エンドポイントが再生成される。これは初期開発だけでなく、避けられない変化を通じてエンジニアリング速度を保つ話だ。

## 中核原則 (Core Principles)

**共通言語としての仕様 (Specifications as the Lingua Franca)**: 仕様が主たる成果物。コードは特定の言語とフレームワークでの表現。ソフトウェア保守とは仕様の進化。

**実行可能な仕様 (Executable Specifications)**: 仕様は、動くシステムを生むのに十分な精密さ・完全性・無曖昧性を持たなければならない。これが意図と実装の間のギャップを消す。

**継続的洗練 (Continuous Refinement)**: 整合性検証は継続的に起こる、ワンタイムのゲートではない。AI は仕様の曖昧性・矛盾・欠落を継続的に分析する。

**リサーチ駆動の文脈 (Research-Driven Context)**: リサーチエージェントが仕様プロセス全体を通じて重要な文脈を集める ― 技術選択肢、性能影響、組織制約を調査する。

**双方向フィードバック (Bidirectional Feedback)**: 本番現実が仕様進化に情報を与える。メトリクス、インシデント、運用学習が仕様洗練の入力になる。

**探索のための分岐 (Branching for Exploration)**: 同じ仕様から複数の実装アプローチを生成し、性能・保守性・UX・コストといった異なる最適化目標を探る。

## 実装アプローチ

現時点で SDD を実践するには、既存ツールを組み合わせ、プロセス全体を通じて規律を保つ必要がある。次のもので実践できる:

- 反復的仕様開発のための AI アシスタント
- 技術的文脈を集めるリサーチエージェント
- 仕様を実装に翻訳するコード生成ツール
- 仕様ファーストのワークフローに適応したバージョン管理
- 仕様文書の AI 分析による整合性チェック

鍵は、仕様を信頼できる情報源として扱い、コードを「仕様に仕える生成出力」として扱うこと ― その逆ではなく。

## コマンドで SDD を効率化する

SDD メソドロジーは、「仕様 → 計画 → タスク」のワークフローを自動化する 3 つの強力なコマンドで大きく強化される:

### `/speckit.specify` コマンド

このコマンドは、シンプルな機能記述（ユーザープロンプト）を、自動的なリポジトリ管理を伴う完全で構造化された仕様に変換する:

1. **自動的な機能番号付け**: 既存仕様をスキャンして次の機能番号を決定（例: 001, 002, 003, …, 1000 ― 3 桁を超えると自動拡張）
2. **ブランチ作成**: 記述からセマンティックなブランチ名を生成し、自動作成
3. **テンプレートベース生成**: 機能仕様テンプレートをコピーし、要件でカスタマイズ
4. **ディレクトリ構造**: `specs/[branch-name]/` 構造を作る

### `/speckit.plan` コマンド

機能仕様ができたら、このコマンドが包括的な実装計画を作る:

1. **仕様分析**: 機能要件、ユーザーストーリー、受け入れ基準を読み解く
2. **コンスティチューション準拠**: プロジェクトのコンスティチューション (constitution) とアーキテクチャ原則との整合を確保
3. **技術翻訳**: ビジネス要件を技術アーキテクチャと実装詳細に変換
4. **詳細ドキュメント**: データモデル、API 契約、テストシナリオの補助ドキュメントを生成
5. **クイックスタート検証**: 主要な検証シナリオを記述したクイックスタートガイドを生成

### `/speckit.tasks` コマンド

計画ができた後、このコマンドが計画と関連設計ドキュメントを分析して、実行可能なタスクリストを生成する:

1. **入力**: `plan.md`（必須）と、あれば `data-model.md`、`contracts/`、`research.md`
2. **タスク導出**: 契約・エンティティ・シナリオを具体的なタスクに変換
3. **並列化**: 独立タスクに `[P]` をマークし、安全な並行グループを示す
4. **出力**: 機能ディレクトリに `tasks.md` を書き出し、タスクエージェントによる実行に備える

### 例: チャット機能を構築する

これらのコマンドが従来の開発ワークフローをどう変えるかの例:

**伝統的なアプローチ:**

```
1. PRD をドキュメントで書く (2-3 時間)
2. 設計書を作る (2-3 時間)
3. プロジェクト構造を手動で設定 (30 分)
4. 技術仕様を書く (3-4 時間)
5. テスト計画を作る (2 時間)
合計: 約 12 時間のドキュメント作業
```

**コマンドを使った SDD アプローチ:**

```
# Step 1: 機能仕様を作成 (5 分)
/speckit.specify Real-time chat system with message history and user presence

# 自動で:
# - ブランチ "003-chat-system" を作成
# - specs/003-chat-system/spec.md を生成
# - 構造化された要件で埋める

# Step 2: 実装計画を生成 (5 分)
/speckit.plan WebSocket for real-time messaging, PostgreSQL for history, Redis for presence

# Step 3: 実行可能なタスクを生成 (5 分)
/speckit.tasks

# 自動的に作られるもの:
# - specs/003-chat-system/plan.md
# - specs/003-chat-system/research.md (WebSocket ライブラリ比較)
# - specs/003-chat-system/data-model.md (Message と User のスキーマ)
# - specs/003-chat-system/contracts/ (WebSocket イベント、REST エンドポイント)
# - specs/003-chat-system/quickstart.md (主要検証シナリオ)
# - specs/003-chat-system/tasks.md (計画から導出したタスクリスト)
```

15 分で次が手に入る:

- ユーザーストーリーと受け入れ基準を持つ完全な機能仕様
- 技術選択と理由付きの詳細な実装計画
- コード生成に使える API 契約とデータモデル
- 自動・手動の両方のテストシナリオ
- すべての文書が機能ブランチでバージョン管理される

### 構造化された自動化の威力

これらのコマンドは時間節約だけでなく、一貫性と完全性を強制する:

1. **見落としがない**: 非機能要件からエラーハンドリングまで、テンプレートがすべての側面を確実にカバー
2. **追跡可能な判断**: すべての技術選択が特定の要件にリンクする
3. **生きたドキュメント**: 仕様がコードを生成するから、ドキュメントとコードが同期する
4. **素早い反復**: 要件を変えて計画を再生成するのが、数日ではなく数分

コマンドは仕様を静的な文書ではなく実行可能な成果物として扱うことで、SDD 原則を体現する。仕様プロセスを「やむを得ない悪」から「開発の駆動力」へ変える。

### テンプレート駆動の品質: 構造が LLM をより良い成果へ制約する仕組み

これらのコマンドの真の力は、自動化だけでなく、テンプレートが LLM の振る舞いをより高品質な仕様へ導く点にある。テンプレートは、LLM の出力を生産的な方向に制約する洗練されたプロンプトとして働く:

#### 1. **過早な実装詳細を防ぐ**

機能仕様テンプレートは明示的にこう指示する:

```
- ✅ Focus on WHAT users need and WHY
- ❌ Avoid HOW to implement (no tech stack, APIs, code structure)
```

この制約により、LLM は適切な抽象度を保つ。「React と Redux で実装する」に飛びつきがちな LLM を、「ユーザーは自分のデータがリアルタイム更新されることを必要としている」に留める。これにより、実装技術が変わっても仕様は安定する。

#### 2. **明示的な不確実性マーカーを強制する**

両テンプレートとも `[NEEDS CLARIFICATION]` マーカーの使用を必須とする:

```
ユーザープロンプトからこの仕様を作るとき:
1. すべての曖昧性をマーク: [NEEDS CLARIFICATION: specific question] を使う
2. 推測しない: プロンプトに指定がなければマークする
```

これは「もっともらしいが誤っているかもしれない仮定を置く」という LLM のよくある振る舞いを防ぐ。「ログインシステム」がメール+パスワード認証だと推測する代わりに、`[NEEDS CLARIFICATION: auth method not specified - email/password, SSO, OAuth?]` とマークさせる。

#### 3. **チェックリストによる構造化された思考**

テンプレートには仕様の「単体テスト」として機能する包括的なチェックリストが含まれる:

```
### Requirement Completeness

- [ ] [NEEDS CLARIFICATION] マーカーが残っていない
- [ ] 要件はテスト可能で曖昧でない
- [ ] 成功基準は測定可能
```

このチェックリストにより、LLM は自身の出力を体系的に自己レビューし、漏れがちなギャップを捉える。LLM に品質保証フレームワークを与えるようなもの。

#### 4. **ゲートを通じたコンスティチューション準拠**

実装計画テンプレートは、フェーズゲートを通じてアーキテクチャ原則を強制する:

```
### Phase -1: Pre-Implementation Gates

#### Simplicity Gate (Article VII)

- [ ] プロジェクト数 ≤ 3 か？
- [ ] 将来対応 (future-proofing) を入れていないか？

#### Anti-Abstraction Gate (Article VIII)

- [ ] フレームワークを直接使っているか？
- [ ] モデル表現は単一か？
```

ゲートは、LLM が複雑さを明示的に正当化することを要求し、過剰設計を防ぐ。ゲートが落ちた場合、LLM は「Complexity Tracking」セクションに理由を文書化しなければならず、アーキテクチャ判断に説明責任が生まれる。

#### 5. **階層的な詳細管理**

テンプレートは適切な情報アーキテクチャを強制する:

```
**重要**: この実装計画は高レベルで読みやすい状態を保つこと。
コードサンプル、詳細なアルゴリズム、広範な技術仕様は、適切な
`implementation-details/` ファイルに置くこと
```

これは「仕様が読めないコードダンプになる」というよくある問題を防ぐ。LLM は適切な詳細度を保ち、複雑さを別ファイルへ切り出しつつメイン文書を航行可能に保つことを学ぶ。

#### 6. **テストファースト思考**

実装テンプレートはテストファースト開発を強制する:

```
### File Creation Order
1. Create `contracts/` with API specifications
2. Create test files in order: contract → integration → e2e → unit
3. Create source files to make tests pass
```

この順序制約により、LLM は実装の前にテスト可能性と契約を考えるようになり、より堅牢で検証可能な仕様につながる。

#### 7. **推測機能を防ぐ**

テンプレートは明示的に推測を抑制する:

```
- [ ] 推測的・「あってもいいかも」の機能はない
- [ ] すべてのフェーズに明確な前提と成果物がある
```

これにより、実装を複雑にする「あったらいいね」機能を LLM が加えるのを止める。すべての機能は、明確な受け入れ基準を持つ具体的なユーザーストーリーにトレースできなければならない。

### 複合効果

これらの制約が一緒に働くことで、仕様は次の性質を持つ:

- **完全 (Complete)**: チェックリストが見落としを防ぐ
- **明確 (Unambiguous)**: 強制された明確化マーカーが不確実性を可視化する
- **テスト可能 (Testable)**: テストファースト思考がプロセスに組み込まれている
- **保守可能 (Maintainable)**: 適切な抽象度と情報階層
- **実装可能 (Implementable)**: 具体的な成果物を持つ明確なフェーズ

テンプレートは LLM を「創造的なライター」から「規律ある仕様エンジニア」に変え、その能力を「一貫して高品質で実行可能な仕様」を生むことに向ける。

## コンスティチューションの基盤 — アーキテクチャの規律を強制する

SDD の中心にはコンスティチューション (constitution) — 仕様がコードになるための不変原則の集合 — がある。コンスティチューション (`memory/constitution.md`) はシステムのアーキテクチャ DNA として働き、生成される実装が一貫性・単純性・品質を保つことを保証する。

### 開発を律する 9 つの条項

コンスティチューションは、開発プロセスのあらゆる側面を形作る 9 つの条項を定める:

#### 第 I 条: ライブラリファースト原則

すべての機能は独立したライブラリとして始まらなければならない — 例外なし。これはモジュラー設計を最初から強制する:

```
Specify のすべての機能は、独立したライブラリとして存在を始めなければならない。
最初に再利用可能なライブラリコンポーネントへ抽象化されることなく、
アプリケーションコード内に直接実装される機能はあってはならない。
```

この原則により、仕様はモノリシックなアプリケーションではなく、モジュラーで再利用可能なコードを生む。LLM が実装計画を生成する際、機能を明確な境界と最小限の依存関係を持つライブラリとして構造化しなければならない。

#### 第 II 条: CLI インターフェース義務

すべてのライブラリはコマンドラインインターフェースで機能を公開しなければならない:

```
すべての CLI インターフェースは:
- テキストを入力として受け取る (stdin、引数、ファイル経由)
- テキストを出力として生む (stdout 経由)
- 構造化データ交換のために JSON フォーマットをサポート
```

これは可観測性とテスト容易性を強制する。LLM は不透明なクラスの中に機能を隠せない — すべてはテキストベースのインターフェースを通じてアクセス・検証可能でなければならない。

#### 第 III 条: テストファースト命令

最も変革的な条項 — テストより先のコードはなし:

```
これは交渉不可: すべての実装は厳格なテスト駆動開発に従わなければならない。
以下より前に実装コードを書いてはならない:
1. 単体テストが書かれている
2. テストが検証されユーザーに承認されている
3. テストが「失敗する」ことが確認されている (Red フェーズ)
```

これは伝統的な AI コード生成を完全に逆転させる。コードを生成して動くことを祈る代わりに、LLM はまず振る舞いを定義する包括的なテストを生成し、承認を得て、そのうえで実装を生成しなければならない。

#### 第 VII 条・第 VIII 条: 単純性とアンチ抽象化

ペアになったこの 2 条項は過剰設計と戦う:

```
Section 7.3: 最小プロジェクト構造
- 初期実装の最大プロジェクト数は 3
- それ以上のプロジェクトは文書化された正当化が必要

Section 8.1: フレームワーク信頼
- フレームワーク機能をラップせず直接使う
```

LLM が当然のように手の込んだ抽象化を作りがちな場面で、これらの条項は複雑さの層ごとに正当化を要求する。実装計画テンプレートの「Phase -1 Gates」がこれを直接強制する。

#### 第 IX 条: インテグレーションファーストテスト

孤立したユニットテストより実世界のテストを優先する:

```
テストは現実的な環境を使わなければならない:
- モックよりも実 DB を優先
- スタブよりも実サービスインスタンスを優先
- 実装前の契約テストは必須
```

これにより、生成コードが理論だけでなく実践でも動くことを保証する。

### テンプレートによるコンスティチューション強制

実装計画テンプレートは、具体的なチェックポイントでこれらの条項を運用化する:

```
### Phase -1: Pre-Implementation Gates

#### Simplicity Gate (Article VII)

- [ ] プロジェクト数 ≤ 3 か？
- [ ] 将来対応を入れていないか？

#### Anti-Abstraction Gate (Article VIII)

- [ ] フレームワークを直接使っているか？
- [ ] モデル表現は単一か？

#### Integration-First Gate (Article IX)

- [ ] 契約は定義されているか？
- [ ] 契約テストは書かれているか？
```

これらのゲートはアーキテクチャ原則のコンパイル時チェックとして働く。LLM はゲートを通過するか、「Complexity Tracking」セクションに正当化された例外を文書化するかのどちらかをしない限り、先へ進めない。

### 不変原則の力

コンスティチューションの力は不変性 (immutability) にある。実装詳細は進化できるが、コア原則は不変だ。これにより:

1. **時間を超えた一貫性**: 今日生成されるコードは、来年生成されるコードと同じ原則に従う
2. **LLM 間の一貫性**: 異なる AI モデルでもアーキテクチャ的に互換なコードが生まれる
3. **アーキテクチャの一貫性**: すべての機能がシステム設計を強化する（損なうのではなく）
4. **品質保証**: テストファースト、ライブラリファースト、単純性の原則が保守可能なコードを保証

### コンスティチューションの進化

原則は不変だが、その適用は進化できる:

```
Section 4.2: Amendment Process
このコンスティチューションの修正には以下が必要:
- 変更理由の明示的な文書化
- プロジェクトメンテナーによるレビューと承認
- 後方互換性評価
```

これによりメソドロジーは安定性を保ちつつ学習・改善できる。コンスティチューション自身が、日付付き修正による進化を示しており、実世界の経験に基づいて原則を精錬する例を示している。

### ルールを超えて — 開発哲学として

コンスティチューションは単なるルールブックではない。LLM がコード生成を考える方法を形作る哲学だ:

- **不透明性より可観測性**: すべては CLI を通じて検査可能でなければならない
- **賢さより単純さ**: シンプルに始め、必要が証明されたときだけ複雑さを足す
- **孤立より統合**: 人工環境ではなく実環境でテストする
- **モノリスよりモジュール**: すべての機能は明確な境界を持つライブラリ

これらの原則を仕様と計画のプロセスに埋め込むことで、SDD は生成コードが単に機能するだけでなく、保守可能で、テスト可能で、アーキテクチャ的に健全であることを保証する。コンスティチューションは AI を「コード生成器」から「システム設計原則を尊重し強化するアーキテクチャ的パートナー」へ変える。

## 変革 (The Transformation)

これは開発者を置き換えたり、創造性を自動化したりする話ではない。機械的な変換を自動化することで、人間の能力を増幅する話だ。仕様、リサーチ、コードが一緒に進化する密なフィードバックループを作り、各反復が意図と実装の間のより深い理解とより良い整合をもたらすことだ。

ソフトウェア開発には、意図と実装の整合を保つためのより良いツールが必要だ。SDD は、コードを単に導くのではなく生成する実行可能な仕様を通じて、この整合を達成するメソドロジーを提供する。
